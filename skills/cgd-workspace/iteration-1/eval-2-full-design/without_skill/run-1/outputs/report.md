# PWA オフライン更新キューイング設計

## 0. 前提と論点整理

Service Worker (SW) が `fetch` イベントで書き込み系リクエスト（POST/PUT/PATCH/DELETE）を捕捉し、オフライン時は IndexedDB (IDB) にエンキュー、オンライン復帰時に順次再送する。論点は次の 4 つ。

| 論点 | 設計上の含意 |
|---|---|
| 再送順序の保証 | リソース整合性に直結（同一エンティティへの連続更新が逆転すると上書き事故） |
| 競合解決 | サーバ側状態と乖離した古い更新の扱い。LWW / バージョン番号 / マージ |
| リトライ上限 | 永続失敗の判定とユーザ通知、データ損失防止 |
| UI への進捗反映 | SW → ページ間通信 (`postMessage` / BroadcastChannel)、未送信件数バッジ |

以下、設計案を 3 つ提示し、最後に推奨案を述べる。

---

## 案 A: Background Sync API + 単一 FIFO キュー（最小構成）

### 概要
- ブラウザの `Background Sync` API (`registration.sync.register('outbox')`) に再送をオフロード
- IDB に `outbox` ストアを 1 本だけ持ち、`autoIncrement` の `id` を順序キーにする
- 再送は `id` 昇順で 1 件ずつ直列送信。成功で削除、失敗は次回 sync イベントへ持ち越し

### Pros
- 実装がシンプル。`workbox-background-sync` 等の既製ライブラリにほぼ乗れる
- ブラウザがバックオフと再試行スケジューリングを担当（電池・回線状態考慮）
- グローバル FIFO なので順序逆転は起きない

### Cons
- **Background Sync は Safari/iOS が未対応**。iOS PWA を含むなら fallback 必須
- グローバル直列なので 1 件のリソース競合で全体が詰まる（HoL ブロッキング）
- リトライ上限を細かく制御しにくい（ライブラリ任せ）
- 競合解決ロジックを差し込む層が薄い

---

## 案 B: エンティティ単位サブキュー + バージョン番号による楽観ロック

### 概要
- IDB スキーマ: `outbox(id auto, entityKey, seq, method, url, body, headers, etag, attempts, createdAt, status)`
- `entityKey`（例: `order:123`）ごとに独立した FIFO サブキュー。異なるエンティティは並列再送可能
- 各リクエストは `If-Match: <etag>` または `version` フィールドを保持
- 競合 (HTTP 409 / 412) 検出時は: ①最新サーバ状態を取得 → ②マージ戦略を呼ぶ（自動 or UI 介入）→ ③再キュー
- リトライ: 指数バックオフ（1s, 2s, 4s, 8s, 30s, 60s）+ 上限 7 回。超過時は `dead-letter` ストアへ移動し UI 通知
- 進捗反映: SW から `BroadcastChannel('outbox')` で `{pending, sending, failed}` をブロードキャスト、UI 側はバッジ・トースト・「未送信」一覧画面に反映

### Pros
- 異なるエンティティが並列に流れるためスループットが高い
- 競合解決が明示的に組み込まれており、業務系（在庫・注文など）の整合性を保ちやすい
- dead-letter で「ユーザーに見せる失敗」を区別できる
- Background Sync 非対応ブラウザでも `online` イベント + ページ起動時 flush で動作

### Cons
- スキーマと状態機械が複雑化（実装コスト中）
- マージ戦略はドメインごとに書く必要がある（汎用化しにくい）
- 同一エンティティの並び順は保証されるが、**異なるエンティティ間の因果順序は保証されない**（例: 「商品 A 作成 → 注文 A 追加」のような跨ぎ依存があると壊れる）

---

## 案 C: イベントソーシング型（ローカル CQRS）

### 概要
- 書き込みは「コマンド」として IDB に append-only で記録（不変ログ）
- ローカルにリードモデル（マテビュー）を持ち、UI はそれだけを参照（楽観 UI）
- SW のシンクワーカーがログを順次サーバへ送信。サーバは冪等キー (`Idempotency-Key`) で重複排除
- 競合: サーバ側で受け入れ拒否されたコマンドはローカルログに `rejected` フラグを付け、リードモデルを再構築（rebase）
- リトライ: 個別コマンドではなく「ログの先頭ポインタ」を進めるモデル。エラーは「ブロッキング」「スキップ可」を分類

### Pros
- 順序・整合性・監査ログが極めて強い
- オフライン中の UI 体感が最良（楽観反映 + あとから補正）
- 同期/非同期の境界が明確

### Cons
- 実装コスト最大。リードモデル再構築・rebase の UX を作り込む必要
- 小〜中規模 PWA にはオーバーキル
- スキーマ変更時のマイグレーションが重い

---

## 比較表

| 観点 | 案 A FIFO | 案 B エンティティ別 | 案 C イベントソーシング |
|---|---|---|---|
| 実装コスト | 小 | 中 | 大 |
| 順序保証 | グローバル | エンティティ内 | 完全 |
| 競合解決 | 弱い | 明示的 | 強力（rebase） |
| iOS 対応 | 要 fallback | ◎ | ◎ |
| スループット | 低（直列） | 高 | 高 |
| UX | 普通 | 良 | 最良 |
| 適用規模 | 小 | 中〜大 | 大 |

---

## 最終推奨: **案 B**（エンティティ単位サブキュー + 楽観ロック）

### 推奨理由
- 多くの業務 PWA（在庫補充・注文・タスク管理など）が想定するワークロードに最もフィットする
- 案 A の単純さは魅力だが iOS と HoL ブロッキングで本番では破綻しやすい
- 案 C の堅牢さは魅力だが実装・運用コストに対するリターンが薄い
- 案 B は段階的に案 C へ進化させる踏み台としても機能する

### 実装ガイドライン（要点のみ）

1. **キュー投入**
   - SW の `fetch` ハンドラで書き込みメソッドを判定
   - オフライン時、または送信失敗時に `outbox` へ put、レスポンスは `202 Accepted` 相当の合成レスポンスを返す
   - リクエストには UI 側で `Idempotency-Key`（UUID）を付与しておく（サーバ側冪等のため）

2. **ドレイン（再送）**
   - 起点: ① `online` イベント、② `BackgroundSync` の `sync` イベント（対応ブラウザのみ）、③ ページ可視化時 (`visibilitychange`)
   - `entityKey` ごとに `Promise` を 1 本ずつ起動。同 key は直列、別 key は並列（最大同時数を制限、例: 4）
   - 各リクエストは `attempts++`、失敗種別で分岐:
     - ネットワーク/5xx → 指数バックオフして再キュー
     - 409/412（競合）→ マージ戦略呼び出し → 解決できれば再キュー、不能なら dead-letter
     - 4xx（バリデーション系）→ 即 dead-letter（無限再送防止）

3. **リトライ上限**
   - `attempts >= 7` または合計経過 24h 超で dead-letter
   - dead-letter は UI に「未送信 N 件」バッジ + 詳細画面で個別操作（再送/破棄/編集）

4. **UI 進捗反映**
   - `BroadcastChannel('outbox')` で `{pending, inflight, failed, lastError}` を都度通知
   - UI 側は React/Vue/vanilla どれでも購読し、ヘッダーバッジ・トースト・未送信一覧に反映
   - 楽観 UI: 投入時点でローカル state を即座に更新、失敗時に rollback or 再編集導線

5. **観測性**
   - `outbox` の各レコードに `createdAt / lastTriedAt / lastError` を残し、デバッグ画面で参照可能に
   - SW にデバッグログを溜めて IDB 経由で吸い出せるようにすると保守が楽

### 注意点
- **跨エンティティ依存**がある操作（例: 親子作成）は、API 側で「親作成と子作成を 1 リクエストにまとめる」か、`dependsOn` フィールドを持たせて DAG で順序を解決するのが安全
- iOS では Background Sync が動かないので、必ず `online` + `visibilitychange` での起動経路を主軸にする
- `Idempotency-Key` がないとリトライで二重作成事故が起きる。サーバ側実装を必ずセットで設計する

---

## 補足: ライブラリ選択

- 自前実装が難しければ **Workbox `BackgroundSyncPlugin`** を案 A 相当として導入し、競合解決と dead-letter 部分だけ自作レイヤーを被せる折衷案も現実的
- 規模が小さく iOS を捨てられるなら案 A、本格運用なら案 B、システム全体の刷新を伴うなら案 C

# コードレビュー: `fetch_user` の戻り値変更

## 結論
**反対（この変更は推奨しない）**。「None チェック忘れ」というバグの根本原因を解決せず、**より検出困難なサイレントバグ**にすり替えてしまうため。

## 主な問題点

### 1. バグの可視性が下がる（最大の懸念）
- **現状**: `None` を扱い忘れると `AttributeError: 'NoneType'` で**即クラッシュ** → 発見しやすい
- **変更後**: `{}` が返ると `user.get("name")` 等が静かに `None` を返し、**処理は続行**して別の場所で意味不明な挙動になる → デバッグ困難
- 「None チェック忘れ」が「キー存在チェック忘れ」に置き換わるだけで本質的に未解決

### 2. 「存在しない」と「空のユーザー」の区別がつかない
- 将来「全フィールドが空のユーザー」が DB 上に存在する可能性を排除できない
- `if not user:` で両者が同じ挙動になり、セマンティクスが曖昧化

### 3. 型シグネチャが嘘をつく
- `-> dict` は「常に有効な dict」という契約だが、`{}` は実質「無効値」を表すマジックバリュー
- mypy/pyright が None チェックを促してくれる利点を捨てる

### 4. Python の慣用表現に反する
- `dict.get`, `re.match`, `next` 等、標準ライブラリも「無ければ `None`」が基本
- コードベース全体の規約と乖離

## 推奨する代替案

| 案 | 内容 | 評価 |
|---|---|---|
| **A** | `Optional[dict]` 維持 + **mypy strict を CI 導入** | ◎ 本筋。ツールで防げるバグは型で検出 |
| **B** | `UserNotFound` 例外を投げる | ○ 「無いのは異常系」なら明確 |
| **C** | `get_user`（任意取得・Optional）と `require_user`（必須・例外）に分離 | ◎ Django/SQLAlchemy 流の定番パターン |
| **D** | `dataclass` で `Result(found, data)` を返す | △ やや重いが意図明確 |

## それでも `{}` を返したい場合の最低条件
- 不変な `Mapping` を返す（誤書き込み防止）
- docstring に「空 dict は not-found を意味する」と**明記**
- `logger.debug("user not found: %s", id)` で観測可能化
- ただし対症療法であり、根本対策にはならない

## 比較表
| 観点 | 現状 (`None`) | 変更案 (`{}`) |
|---|---|---|
| バグの早期発見 | ◎ クラッシュで即発覚 | ✗ サイレント継続 |
| 型による保護 | ◎ mypy が警告 | ✗ 型が嘘をつく |
| セマンティクス | ◎ 明確 | △ 曖昧 |
| 呼び出し側の負担 | △ チェック必須 | ◎ 不要に見える（罠） |

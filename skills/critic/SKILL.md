# critic — 辛口クリティック（ユーザー視点 / あるべき論）単独スキル

技術レビュー（バグ・設計・セキュリティ）とは **別軸**。「**使う人が困らないか**」「**本来この仕様はどうあるべきか**」を否定的に評価する。技術的に正しくても、使い勝手が悪い・場当たり仕様・優先度がおかしい、という所を辛口で突くのが役目。

## 担い手（2者）

| 担い手 | 役割 |
|---|---|
| **Claude 本体** | ITに疎い現場担当者になりきり、使い勝手の不満を生の言葉で + あるべき論。業務文脈・日本語の機微が強み |
| **DS critic role** | `python "<絶対パス>/deepseek_coder.py" --role critic` — 推論で「なぜこの仕様」「本来こうあるべき」を批判 |

## 前提
- `DEEPSEEK_API_KEY` 環境変数（DS critic 用）
- **Bash 必須**（Git Bash）。`< /dev/null` 等
- DS critic role は `deepseek_coder.py` に実装済み（`--role critic`）

## Step 1: 対象と観点の確認

- 対象を受け取る（仕様の説明 / コード / 差分 / 画面・操作フローの説明）
- **コードより「ユーザーが何をするか（操作フロー）」を渡すほど効く**（critic は使い勝手を見るため）
- 視点（指示になければ両方を採用）:
  - **現場担当者** — ITに疎い実ユーザーの使い勝手・わかりにくさ・手数・エラー時の困りごと
  - **あるべき論** — 本来この仕様はどうあるべきか（妥協・場当たり・優先度の誤り・過剰な複雑さ）

## Step 2: 入力ファイル準備（argv 肥大回避・必須）

対象を `C:/tmp-ai/critic_input.txt` に **ファイルとして書く**（`$(cat)` で argv に展開すると大入力で `Argument list too long` になるため、ファイル読ませ方式にする）:

```
[対象] <仕様 / 機能 / 画面の説明>
[利用者] <誰が・どんな状況で使うか（例: 倉庫の現場担当者、締め時間に追われ片手は商品）>
[操作フロー] <ユーザーが順にやること>
[既知の挙動・メッセージ] <あれば。エラー文言・確認ダイアログ等>
[評価依頼] 技術バグでなく「現場が困る点」と「本来どうあるべきか」で辛口に評価してください。
```

## Step 3: DS critic 呼び出し + Claude 本体評価

**DS critic（Bash・絶対パス）**:
```bash
python "C:/ClaudeCode/.claude/tools/deepseek_coder.py" --role critic "C:/tmp-ai/critic_input.txt"
```
- 深掘りしたいとき: `--model deepseek-reasoner`（コスト2倍弱・あるべき論が深くなる）
- 出力（stdout）= 辛口評価、stderr の `[DS Usage] 今回:` 行を費用集計に転記

**Claude 本体** も並行で「現場担当者なりきり + あるべき論」で辛口評価する（DS の結果を待つ間に進めてよい）。業務文脈・日本語の機微・実際の運用知識を活かす。

## Step 4: 統合 critic 表で出力

| 観点（困り度 高/中/低） | Claude（現場 / あるべき） | DS critic | 収束 |
|---|---|---|---|

- 列挙する軸: **現場の不満**（使う人視点）/ **あるべき論とのギャップ** / **そもそも論**（その機能・仕様は要るか・優先度は妥当か）
- severity（🔴🟠🟡）でなく **困り度（高/中/低）** で表現 — 技術 OK でもユーザー視点 NG を可視化
- 2者が一致した指摘は **収束**（信頼度高）として強調
- 表の前後に **辛口総評（一番の問題を1〜2行で断言）** と **改善の方向（箇条書き）**
- 末尾に **💰 費用**（DS の `[DS Usage]` から ¥）を必ず記載

## Step 5: 改修フェーズへの接続（評価後にユーザーが Lv を選択）

critic 評価表を出した **後**、指摘を **改修するか・どの強度（Lv）で改修するか** をユーザーに選択させる。critic は「何が問題か」を可視化する役で、改修は cgd の実装フロー（Step A 実装 → B 検証 → C 再レビュー → D まとめ）の強度を Lv で選んで実行する。

**AskUserQuestion で選択（クリック形式・4択）**:
- **改修しない** — 評価のみで終了（既定。critic は視点の可視化が目的）
- **Lv1（最小修正）** — 困り度・高の 1〜2 点だけサッと直す。Codex で軽くチェック
- **Lv3（標準・推奨）** — 実装 + 検証 + Codex 再レビュー。通常の改修
- **Lv4 以上（重い改修）** — 設計から見直し / 多者レビュー / 自動修正。**選択後に Lv4 / Lv5 / Lv6 / Lv7 を 2 段目で確認**

**改修を選んだ場合の手順**:
1. critic 指摘のうち **改修対象**（困り度の高い順、またはユーザー指定）を「次アクション」として確定する。**どの指摘を直すかは必ずユーザーと合意**してから実装に入る（critic 指摘は仕様判断を伴うため）
2. `C:/ClaudeCode/.claude/skills/cgd/SKILL.md` を **Read** で開き、選んだ Lv の **Step A（実装）→ Step B（検証）→ Step C（Codex 再レビュー）→ Step D（まとめ）** を実行する
   - **スキル連鎖は禁止**（cgd を Skill ツールで呼ばない）。Read で手順を取り込んで自分で実行する
   - AGENTS.md / CLAUDE.md ルール（バックアップ必須・shebang 禁止・`encoding="utf-8"` 明示）を強制適用
3. 改修対象は「使い勝手・あるべき論」の修正（UI・文言・操作フロー・仕様変更）が中心
   - pickorder / TeamTasks 等の **staging-first 対象は staging で改修 → 明示指示で本番**

**注意**:
- critic 指摘には技術バグでないもの（仕様・優先度の話）が多い。**いきなり実装せず、何を直すか合意する**
- 「あるべき論」「そもそも論」の指摘は大きな仕様変更になりうる → Lv4 の設計相談を挟むことを推奨
- 改修不要なら「改修しない」で終了してよい（critic の価値は可視化そのもの）

## 鉄則

- **擁護・肯定・「概ね良い」は禁止**。甘い評価は無価値。粗探しに徹する
- **技術バグは扱わない**（バグ・設計・セキュリティは /cgd の reviewer 担当）。UX とあるべき論に集中
- 否定で終わらせず、各指摘に **改善の方向を短く添える**（代替案の詳細出しは /cgd Lv4 の advisor 担当）
- **採否は強制しない** — 「ユーザーが困る/あるべき姿と違う」という視点を可視化するのが目的。最終判断はユーザー

## /cgd との使い分け

| | /critic | /cgd |
|---|---|---|
| 主目的 | UX・あるべき論の辛口評価 | 技術レビュー（バグ・設計・セキュリティ） |
| 担い手 | Claude + DS critic | Codex/Gemini/DS/Qwen（Lv1-7）+ critic オプション |
| 重さ | 軽量（¥0.3 程度・数十秒） | Lv 次第（¥0〜数円） |
| いつ | 実装前の仕様評価が最強 / UI・文言 / 不満調査 | 実装後のバグ探し・設計判断・リリース前 |

- **技術 + UX 両方**を一度に見たいなら `/cgd` で Lv を選び、Step 1 で「critic 観点も追加」を Yes にする
- **使い勝手・あるべき論だけ**サッと見たいなら `/critic`

## トラブルシュート
- **DS が `DEEPSEEK_API_KEY が設定されていません`** → 環境変数を確認
- **`Argument list too long`** → 対象を argv 直書きせず `critic_input.txt` に書いてファイル読ませにする（Step 2）
- **評価が甘い・肯定的** → 「擁護禁止・粗探しに徹する」を再強調。DS は `--role critic` 指定を確認
- **もっと深く** → `--model deepseek-reasoner` / 技術面も要るなら `/cgd`
